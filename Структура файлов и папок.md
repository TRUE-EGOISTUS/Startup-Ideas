# Структура проекта: описание папок и файлов
## 1. Общая структура

project_for_startup/
├── backend/          # серверная часть (FastAPI)
├── frontend/         # клиентская часть (React)
├── docker-compose.yml  # конфигурация для локального запуска (Docker)
├── .env.example      # пример переменных окружения
├── .gitignore        # исключения для Git
├── Cтруктура файлов и папок  # описание структуры папок
└── Описание идеи.md        # общее описание проекта

* backend/ – весь код серверной части (FastAPI).

* frontend/ – весь код клиентской части (React).

* docker-compose.yml – конфигурация для запуска всех сервисов (база данных, бэкенд, фронтенд) одной командой.

* .env.example – пример файла с переменными окружения (секреты, настройки). Каждый разработчик копирует его в .env и заполняет своими данными.

* .gitignore – список файлов и папок, которые не попадают в репозиторий (например, .env, node_modules, __pycache__).

* Описание идеи.md – описание проекта, точнее идеи стартап-проекта

### Пояснения
* app/main.py – точка входа. Здесь создаётся экземпляр FastAPI, подключаются роутеры, настраиваются CORS, обработчики ошибок.

* app/config.py – загрузка переменных окружения с помощью Pydantic Settings. Все настройки (секреты, URL базы данных, таймауты) хранятся здесь.

* app/dependencies.py – общие зависимости, которые используются во многих эндпоинтах: получение текущего пользователя, проверка прав, сессия базы данных.

* core/ – базовые компоненты, которые используются во всём приложении.

    * database.py – настройка SQLAlchemy (engine, sessionmaker, Base).

    * security.py – хеширование паролей (bcrypt), создание и валидация JWT-токенов.

    * exceptions.py – кастомные классы исключений (например, NotFoundException, PermissionDenied), чтобы в роутерах их перехватывать и возвращать понятный HTTP-ответ.

    * utils.py – утилиты (генерация случайного кода, форматирование дат, валидация slug и т.п.).

* models/ – SQLAlchemy модели (классы, описывающие таблицы в PostgreSQL).

    * user.py – модели User, SpecialistProfile, CompanyProfile.

    * task.py – модели Task, TaskResponse, TaskExecution, Review.

    * idea.py – модели Idea, IdeaResponse, Project.

    * chat.py – модель Message.

    * notification.py – модель Notification.

* schemas/ – Pydantic-схемы для валидации входящих данных и сериализации ответов. Обычно делятся на Create..., Update..., Read....

* api/ – эндпоинты (роутеры).

    * v1/ – версия API. Внутри каждый файл отвечает за свою группу:

        * auth.py – регистрация, логин, logout, GitHub OAuth.

        * users.py – просмотр/редактирование профиля, рейтинг, история выполненных задач.

        * tasks.py – CRUD задач, лента, отклики, выполнение, приёмка, оценки.

        * ideas.py – публикация идей, отклики, формирование команды.

        * projects.py – создание приватного проекта, управление участниками.

        * chat.py – эндпоинты для long polling (получение и отправка сообщений).

        * files.py – загрузка аватарок, логотипов, вложений в S3.

        * deps.py – зависимости, специфичные для API (например, проверка, что пользователь является компанией).

* services/ – бизнес-логика. Роутеры вызывают функции из services, передавая им данные, а те возвращают результат. Здесь нет прямого доступа к HTTP, только к моделям и репозиториям.

* repositories/ – (опционально) слой абстракции для работы с базой данных. Если проект растёт, он помогает держать запросы к БД в одном месте. Для MVP можно обойтись без него, вызывая методы SQLAlchemy прямо в сервисах.

* integrations/ – всё, что взаимодействует с внешними сервисами.

    * github.py – OAuth, получение списка репозиториев, импорт портфолио.

    * email.py – отправка писем через Resend или SMTP.

    * storage.py – загрузка/удаление файлов в S3-совместимое хранилище.

* migrations/ – Alembic миграции. Каждая миграция описывает изменения в схеме БД (добавление/удаление таблиц, колонок и т.д.).

* tests/ – тесты (pytest). Рекомендуется создавать подпапки test_api/, test_services/ и т.д., зеркалирующие структуру app.

* requirements.txt – зависимости для продакшена.

* requirements-dev.txt – дополнительные зависимости для разработки (pytest, black, flake8, pre-commit).

* Dockerfile – инструкция для сборки Docker-образа бэкенда.

* .env – локальный файл с переменными окружения (не попадает в репозиторий).

### Пояснения
* public/ – статические файлы, которые не обрабатываются Vite: index.html, favicon.ico, возможно, глобальные изображения.

* src/ – весь исходный код приложения.

    * main.tsx – точка входа: рендерит App в DOM.

    * App.tsx – корневой компонент: обёртка с роутером, провайдерами (например, для сторов или темы).

    * assets/ – изображения, шрифты, глобальные стили (например, index.css с Tailwind).

    * components/ – переиспользуемые компоненты, сгруппированные по функциональным областям:

        * common/ – универсальные кнопки, инпуты, модалки, лоадеры.

        * layout/ – шапка, подвал, боковое меню, обёртка страницы.

        * task/ – компоненты, связанные с задачами (карточка, форма создания, фильтр).

        * idea/ – компоненты для идей.

        * profile/ – отображение и редактирование профиля.

        * chat/ – окно чата, список сообщений, поле ввода.

    * pages/ – компоненты-страницы, каждый соответствует определённому маршруту. Они собирают вместе компоненты из components и хуки, но не содержат сложной логики.

    * hooks/ – кастомные хуки, которые инкапсулируют бизнес-логику и вызовы API. Например, useTasks возвращает список задач, функции для отклика и т.п.

    * services/ – функции для взаимодействия с API. api.ts – настроенный экземпляр axios (добавляет токен в заголовки, обрабатывает ошибки). Остальные файлы содержат конкретные запросы (например, tasks.ts – функции getTasks, createTask, submitResponse).

    * store/ – глобальное состояние (Zustand). Хранит данные, которые нужны многим компонентам: текущий пользователь, список задач, уведомления.

    * types/ – TypeScript-типы и интерфейсы, импортируемые в компонентах и хуках. Лучше всего синхронизировать их с Pydantic-схемами бэкенда.

    * utils/ – вспомогательные функции без побочных эффектов: форматирование дат, валидация форм, работа с localStorage.

    * routes.tsx – определение маршрутов (например, с помощью react-router-dom). Здесь же можно реализовать защищённые маршруты, доступные только авторизованным пользователям.

* package.json – список зависимостей и скриптов (например, npm run dev, npm run build).

* vite.config.ts – конфигурация Vite: настройка прокси для API (чтобы во время разработки не было CORS), алиасы для путей (@/components и т.п.).

* tailwind.config.js – настройки Tailwind (если вы выбрали его).

* Dockerfile – для контейнеризации фронтенда (обычно многоступенчатая сборка с npm run build и nginx).

* .env – переменные окружения для фронтенда (например, VITE_API_URL).

## 4. Почему именно такая структура
* _Разделение ответственности_ – каждому слою (модели, схемы, бизнес-логика, API, интеграции) отведено своё место. Это упрощает тестирование и модификацию.

* _Масштабируемость_ – вы можете легко выделить микросервис из какого-то модуля, если потребуется.

* _Комфорт для команды_ – все знают, где искать код для задач, для идей, для интеграций. Единообразие уменьшает количество ошибок.

* _Английские названия_ – общеприняты в индустрии, не вызывают проблем с импортами и автодополнением.

Эта структура уже учитывает все особенности вашего проекта: гибрид микро-задач и команд, два режима выполнения задач, long polling для чатов, интеграцию с GitHub, хранение файлов в S3. Она готова к разработке и дальнейшему развитию.
