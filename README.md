# Идея стартапа: Гибридная платформа для выполнения микро-задач и формирования команд стартапов.

## 1. Уточнённое (операционализированное) ценностное предложение
_Для" начинающих IT-специалистов (студенты, выпускники курсов, самоучки)
в ситуации поиска первого практического опыта и возможности участвовать в реальных проектах, но сталкивающихся с отсутствием портфолио и замкнутым кругом «нет опыта – нет работы»
решение обеспечивает выполнение реальных микро-задач от компаний и стартапов с последующей фиксацией результатов в профиле, а также поиск единомышленников для запуска собственных проектов
улучшая время получения подтверждённого опыта с нескольких месяцев до 2–4 недель и объективность оценки навыков через выполненные задачи, а не только резюме
по сравнению с фриланс-биржами Upwork, Kwork, где доминируют опытные исполнители, и классическими досками вакансий Habr Career, не дающими возможности продемонстрировать реальные компетенции
при условиях наличия базовых технических навыков в рамках выбранной специализации и готовности выполнять задачи в установленные сроки._ 
### Почему пользователь поверит:
1. Каждая выполненная задача оставляет в профиле проверяемый артефакт (ссылка на репозиторий, скриншоты) и оценку заказчика – объективное подтверждение навыков.
2. Система рейтинга и отзывов делает профиль прозрачным, снижая риск недобросовестного сотрудничества.
3. Поэтапный трек развития (задачи → проекты → собственный стартап) мотивирует оставаться на платформе.

## 2. Пользовательские сценарии (user flow)
### Основной сценарий: «Специалист находит и выполняет задачу»
1. _Триггер:_ Специалист регистрируется на платформе, заполняет профиль (навыки, ссылка на GitHub).
2. _Вход:_ Заходит в ленту открытых задач.
3. _Фильтрация:_ Применяет фильтры по навыкам (например, Python, React) и сложности (Junior).
4. _Выбор задачи:_ Просматривает карточку задачи, описание, срок, наличие вознаграждения.
5. _Отклик:_ Нажимает «Откликнуться», при желании добавляет сопроводительное сообщение.
6. _Ожидание выбора:_ Заказчик рассматривает отклики и выбирает исполнителя.
7. _Получение доступа:_ Получает уведомление о выборе; задача переходит в статус «В работе».
8. _Выполнение:_ Общается с заказчиком во внутреннем чате, отправляет результат (ссылка на репозиторий, файл).
9. _Приёмка:_ Заказчик проверяет результат и принимает работу, оставляя оценку и отзыв.
10. _Фиксация:_ В профиле появляется завершённая задача, рейтинг обновляется, навык добавляется в «подтверждённые».

### Альтернативный сценарий: «Исполнитель дорабатывает решение»
1. _Триггер:_ Заказчик не принимает работу с первого раза, нажимает «Доработать» и оставляет комментарий.
2. _Уведомление:_ Исполнитель получает уведомление о необходимости доработки.
3. _Вход:_ Открывает чат задачи, видит комментарий заказчика.
4. _Корректировка:_ Вносит изменения в результат, отправляет обновлённую версию.
5. _Повторная проверка:_ Заказчик проверяет исправленный вариант.
6. _Приёмка:_ Если всё удовлетворяет, задача завершается с оценкой (возможно, сниженной).
7. _Фиксация:_ Задача попадает в историю с пометкой «принята после доработки», что видно другим заказчикам.

## 3. Карта функций
### Уровень Core (Ядро MVP):
1. _Регистрация и выбор роли (специалист/компания)_: Без регистрации пользователь не может выполнять действия.
2. _Создание профиля с указанием навыков и ссылок на GitHub:_ Позволяет специалисту представить себя, заказчику – оценить.
3. _Публикация задачи (название, описание, навыки, срок):_ Основной способ создания спроса на платформе.
4. _Лента задач с фильтрацией по навыкам, сложности:_ Обеспечивает навигацию и поиск подходящих задач.
5. _Отклик на задачу:_ Позволяет специалисту заявить о готовности выполнить.
6. _Выбор исполнителя из откликов:_ Завершает этап подбора исполнителя.
7. _Внутренний чат для обсуждения задачи:_ Обеспечивает коммуникацию в процессе выполнения.
8. _Приёмка задачи с оценкой и отзывом:_ Завершает цикл и формирует репутацию.
### Уровень Support:
9. _Публикация идеи стартапа:_ Расширяет возможности для формирования команд.
10. _Отклик на идею:_ Позволяет находить единомышленников.
11. _Создание приватного проекта для команды:_ Организует совместную работу над идеей.
12. _Публикация приватных задач (видимы только приглашённым):_ Даёт гибкость компаниям.
13. _Режим открытого выполнения (без предварительного отклика):_ Упрощает участие для специалистов.
14. _Email-уведомления о событиях:_ Повышает вовлечённость.
15. _Интеграция с GitHub OAuth (импорт репозиториев):_ Упрощает регистрацию и наполняет портфолио.
16. _Система рейтинга (средняя оценка):_ Визуализирует надёжность участников.
17. _Возможность оценить заказчика со стороны исполнителя:_ Стимулирует качественное взаимодействие.
### Уровень Later:
18. _Платные подписки и комиссии:_ Монетизация, не нужна на MVP.
19. _Геймификация (уровни, достижения):_ Не критична для доставки ценности.
20. _Расширенная аналитика для компаний:_ Можно отложить до роста базы.
21. _Верификация компаний через юрлицо:_ Усложняет старт, не обязательна.
22. _Мобильное приложение:_ Достаточно адаптивного веба.
23. _Сложная система модерации:_ Можно начать с ручного контроля.

## 5. Нефункциональные требования
| № | Требование | Обоснование / Ограничения |
| - | ---------- | ------------------------- |
| 1 | Производительность: время отклика API для основных операций (просмотр ленты, отправка отклика) должно составлять не более 500 мс при нагрузке до 50 одновременных пользователей. | Обеспечивает комфортную работу в период старта. |
| 2 | Безопасность: защита от XSS, CSRF, SQL-инъекций; все пароли хешируются (bcrypt); JWT токены хранятся в httpOnly cookies. | Критично для доверия пользователей и защиты данных. |
| 3 | Доступность: платформа должна быть доступна 99.5% времени (не более 3.5 часов недоступности в месяц) в рабочие часы (9:00–21:00 МСК). | Стартап не требует 99.99%, но базовая надёжность важна. |
| 4 | Масштабируемость: архитектура (распределённый монолит) должна позволять вынести модули (чат, уведомления) в отдельные сервисы без переписывания ядра. | Подготовка к возможному росту нагрузки. |
| 5 | Адаптивность: веб-приложение должно корректно отображаться на экранах шириной от 320px до 1920px (мобильные, планшеты, десктопы). | Целевая аудитория активно использует мобильные устройства. |
| 6 | Хранение данных: файлы (аватарки, логотипы, вложения) хранятся в S3-совместимом облаке; база данных PostgreSQL с ежедневными автоматическими бэкапами. | Обеспечивает надёжность и возможность горизонтального масштабирования. |
| 7 | Интеграция с GitHub: OAuth-авторизация должна корректно работать и импортировать публичные репозитории в портфолио пользователя. | Ключевая интеграция для наполнения профилей. |
| 8 | Доступность для людей с ограниченными возможностями (базовый уровень): поддержка клавиатурной навигации, контрастные элементы, атрибуты alt для изображений. | Расширяет аудиторию и соответствует современным стандартам. |
| 9 | Локализация: интерфейс и уведомления на русском языке (MVP ориентирован на русскоязычный рынок). | Целевая аудитория – в основном российские студенты и компании. |
| 10 | Журналирование: логирование ключевых действий (создание задачи, выбор исполнителя, завершение) с уровнями INFO/ERROR для отладки и анализа инцидентов. | Необходимо для быстрого исправления проблем на ранних этапах. |

## 6. 6. Риски и приоритеты 
| Риск | Вероятность | Влияние | Меры смягчения |
| ---- | ----------- | ------- | -------------- |
| Проблема «курицы и яйца» (мало задач / мало специалистов) | Высокая | Критическое | 1. Ручное наполнение первыми задачами командой. 2. Партнёрство с вузами (МИРЭА). 3. Бесплатный тариф для компаний на старте. |
| Недоверие между участниками | Средняя | Высокое | 1. Прозрачный рейтинг и история выполненных задач. 2. Верификация компаний через доменную почту. 3. Интеграция с GitHub как объективное подтверждение. |
| Низкое качество выполнения задач | Средняя | Среднее | 1. Система рейтинга, влияющая на доступ к сложным задачам. 2. Возможность доработки и обоснованного отказа. 3. На первых порах ручная модерация сложных задач. | 
| Сложность реализации чата в реальном времени | Низкая (выбрали long polling) | Среднее | Использовать простой long polling, в будущем перейти на WebSockets. |
| Юридические риски (договорные отношения) | Средняя | Среднее | На старте платформа – информационная площадка, ответственность за договорённости между сторонами не несёт. |
