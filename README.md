# Финальная проверка MVP: гибридная платформа микро-задач и команд.
## Шаг 1. Контроль соответствия MVP исходным гипотезам
### Мини-чек согласнованности
| Параметр | Формулировка |
| -------- | ------------ |
| Сегмент | Начинающие IT-специалисты (студенты, выпускники курсов, самоучки); основатели стартапов и компании, ищущие исполнителей |
| Проблема | Новичкам сложно получить первый практический опыт и подтвердить навыки; основателям трудно найти надёжную команду с проверенными компетенциями | 
| Ценность (value proposition) | Единая экосистема: выполнение реальных микро-задач → формирование портфолио и репутации → поиск партнёров и создание стартап-команд |
| Целевое действие на MVP | Успешное выполнение задачи с получением оценки ИЛИ создание команды через идею |
| Что считаем «успехом» | Не менее 70% опубликованных задач завершены с оценкой; сформировано минимум 3 приватных проекта команд за период пилота |
## Ответы на контрольные вопросы
### Каким образом MVP решает заявленную проблему?
Платформа даёт новичкам доступ к реальным задачам от компаний и сообщества. Выполняя их, специалист получает не просто строчку в резюме, а объективную оценку, подтверждённые навыки и публичное портфолио. Для основателей – возможность искать участников не по словам, а по доказанной истории выполненных задач и рейтингу. Таким образом разрываются замкнутый круг «без опыта – нет работы» и проблема «кота в мешке» при наборе команды.
### Соответствует ли продукт первоначальной концепции и гипотезам ЦА?
Полностью соответствует. Реализованы обе ключевые ветки: «специалист» (профиль, портфолио, отклики/решения, рейтинг) и «компания/автор идеи» (публикация задач и идей, отбор исполнителей, приёмка). Подтверждение навыков через реальные задачи и прозрачная система отзывов прямо решают описанные боли ЦА.

### Выводы аудита
#### Можно убрать / перенести в Later:
* Email-уведомления – перенести в пострелиз. На MVP достаточно внутренних уведомлений и одного письма при регистрации.
* GitHub OAuth – оставить только как опцию, не делая обязательным шагом при регистрации. Импорт репозиториев можно временно заменить ручной вставкой ссылок.

#### Стоит упростить:
* Публикация задачи: добавить простой структурированный шаблон ТЗ (заголовок – описание – теги – срок). Это снизит количество некачественных задач и ускорит заполнение.
* Отправка решения в открытом режиме: убрать обязательное сопроводительное сообщение, оставить только поле для ссылки/файла и кнопку «Отправить».

#### Необходимо добавить в Core (обоснование):
* Кнопка «Пожаловаться на задачу» с указанием причины (некорректное ТЗ, спам, нарушение правил). Без этого мусорные задачи разрушают доверие к платформе, а модерация силами команды на старте невозможна. Жалоба позволит быстро скрывать проблемный контент.

## Шаг 3. Корректировка функциональности
### Произведённые изменения
#### 1. Перенос в Later:
* Автоматические email-уведомления (кроме подтверждения email при регистрации). Внутренние уведомления полностью покрывают потребности MVP.
* Автоматический импорт репозиториев GitHub. Пользователи могут добавить ссылки вручную, функционал будет доработан после валидации гипотез.

#### 2. Упрощение:

* Шаблон ТЗ для заказчика: при создании задачи появляются подсказки (что должно быть в описании, как указать критерии приёмки). Это снижает долю некачественных задач без сложной модерации.

* Отправка решения: поле «Сопроводительное сообщение» становится опциональным. Достаточно прикрепить ссылку на репозиторий или файл. Это ускоряет действие и убирает лишний шаг.

#### 3. Добавление в Core:

* Жалоба на задачу. Рядом с каждой задачей в ленте появляется кнопка «Пожаловаться». Пользователь выбирает причину, задача скрывается из общего списка до ручной проверки. Эта мера защищает ценность платформы как источника качественного опыта.

_Графические правки:_
На карточке задачи добавлена иконка флага для жалобы; поле «сопроводительное сообщение» помечено как необязательное. Страница создания задачи содержит разворачиваемые подсказки под полями.

## Шаг 4. Гипотетическая проверка метрик и воронки.
### North Star Metric (NSM)
#### «Количество успешно завершённых задач в неделю».
Именно завершённая задача с оценкой означает, что специалист получил опыт, а компания – результат. Эта метрика напрямую отражает доставку ключевой ценности MVP. После появления достаточного числа команд можно будет ввести дополнительную NSM «число активных приватных проектов».

### Воронка взаимодействия (7 шагов)
1. Посещение платформы (уникальный посетитель)
2. Регистрация (создание аккаунта с выбором роли)
3. Заполнение профиля (добавление минимум 2 навыков)
4. Просмотр задачи / отклик (нажатие «Откликнуться» или «Отправить решение»)
5. Получение задачи в работу (для классического режима: выбор исполнителя; для открытого – отправка решения)
6. Завершение задачи (принятие результата заказчиком и выставление оценки)
7. Возврат на платформу (повторный вход в течение 7 дней после завершения задачи)

 _Целевое действие, соответствующее NSM, происходит на шаге 6._

 ### Таблица ключевых метрик
| Метрика | Зачем нужна | Как измеряем |
| ------- | ----------- | ------------ |
| Конверсия Посетитель → Регистрация	| Оценка привлекательности посадочной страницы / комьюнити	| Google Analytics / счетчик на лендинге |

### Где ожидаются «провалы» и как их диагностировать
* Шаг 3 → 4 (поиск и первый отклик): новичок может не найти подходящую задачу или испугаться требований. Диагностика: анализ логов поиска и времени до первого отклика.
* Шаг 5 → 6 (завершение задачи): исполнитель может пропасть или сделать некачественно. Диагностика: процент задач, перешедших в статус «На доработке» и не завершённых за срок.
* Шаг 6 → 7 (возврат): пользователь получил опыт и ушёл. Диагностика: опрос при неактивности, причины отказа от повторного использования.

## Ответы на вопросы
### Какие шаги нужно предпринять для реального тестирования?
Подготовить финальную версию MVP с внесёнными правками, создать тестовый контент (3-4 задачи, 2 идеи), настроить среду с доменом и SSL, разработать краткий гайд для тестировщиков, провести 10 индивидуальных сессий, собрать и проанализировать ответы по чек-листу.

### Как бы выглядел процесс сбора обратной связи?
После каждого созвона респондент заполняет Google-форму с перечисленными вопросами. Команда ведёт записи во время демонстрации, фиксируя точки заминок и неожиданные действия. Результаты агрегируются в единую таблицу, метрики успеха рассчитываются автоматически. Качественные ответы категоризируются для выявления повторяющихся проблем.
